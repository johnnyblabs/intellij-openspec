## Context

The OpenSpec IntelliJ plugin (v0.2.3) has shipped workflow features — Fast-Forward, Continue, Verify, Sync Specs, Bulk Archive — faster than the compliance guardrails that should accompany them. Two systemic gaps have emerged:

1. **Validation is lenient.** RFC 2119 keywords and scenario clauses are warnings, not errors. Delta spec structure (`## ADDED`, `## MODIFIED`, etc.) is checked for presence but not for completeness of the requirement blocks inside. Content that violates the OpenSpec contract passes validation.

2. **Compliance is invisible.** There is no notification group, no pre-flight check before archive, and no status indicator telling the user whether their change is OpenSpec-compliant or what action is required to become compliant.

**Affected code:** `BuiltInValidator`, `DeltaSpecInspection`, `SpecSyncService`, `OpenSpecArchiveAction`, `ChangeService`, `WorkflowActionPanel`, `OpenSpecNotifier`, `plugin.xml`.

**Constraints:** The plugin must continue to work without the CLI (built-in mode). This change is marked BREAKING because validation will start rejecting content that previously passed.

## Goals / Non-Goals

**Goals:**
- Enforce OpenSpec-compliant validation: RFC 2119 keywords, scenario structure, and delta operation blocks as errors not warnings
- Surface compliance status in the workflow UI so users can see what's compliant, what failed, and what to do about it
- Tighten spec-sync so merged main specs are validated after merge and MODIFIED operations that target nonexistent requirements fail in strict mode

**Non-Goals:**
- Not implementing the OpenSpec CLI's validation — we align our built-in rules with the contract, but the CLI remains the authoritative validator
- Not implementing post-archive policy enforcement (git commit/push, tracker API calls) — that is project-specific workflow automation, not a plugin feature
- Not changing the artifact DAG structure or adding new artifact types
- Not implementing a compliance dashboard or historical compliance audit — observability is limited to current-change status

## Decisions

### 1. Promote validation severity from WARNING to ERROR for contract-required rules

**Decision:** Change `BuiltInValidator` so that RFC 2119 keywords in requirements and GIVEN/WHEN/THEN clauses in scenarios produce ERROR-level `ValidationResult` entries rather than WARNING. Add structural validation for delta operation sections: each `### Requirement:` block under `## ADDED` / `## MODIFIED` must contain a description and at least one `#### Scenario:` with WHEN/THEN. `## REMOVED` blocks must contain `**Reason**` and `**Migration**`.

**Rationale:** The OpenSpec contract treats these as normative. The plugin's lenient treatment has let malformed specs pass and accumulate. Specs generated by the plugin's own AI pipeline already comply, so this primarily catches manual edits and external tool output.

**Alternatives considered:**
- *Add a "compliance mode" toggle separate from strict mode*: Rejected — compliance is not optional when the goal is contract alignment. Strict mode already exists for warnings-as-errors; this change makes contract violations errors by default.
- *Delegate all validation to the CLI*: Rejected — the plugin must work without the CLI. Built-in rules are the fallback and must be sufficient for contract compliance.

### 2. Introduce a ComplianceService as a project-level service

**Decision:** Create `ComplianceService` (`@Service(Service.Level.PROJECT)`) that computes a `ComplianceStatus` for a change. The status aggregates: (a) artifact completeness, (b) validation results (zero errors), and (c) spec-sync readiness (delta specs exist and target valid capabilities). The service exposes a `checkCompliance(Change): ComplianceResult` method that returns a structured result with per-category pass/fail and actionable remediation messages.

**Rationale:** Compliance checks are currently scattered: `BuiltInValidator` handles spec format, `VerificationService` handles artifact completeness, `SpecSyncService` handles delta parsing. No single component answers "is this change ready to archive in an OpenSpec-compliant way?" A dedicated service composes these existing services into a unified check.

**Alternatives considered:**
- *Extend VerificationService*: Rejected — VerificationService checks artifact completeness and task progress. Compliance is a broader concept (validation + sync readiness). Mixing concerns would bloat the existing service.
- *Inline checks in OpenSpecArchiveAction*: Rejected — the action class should delegate to a service, not contain business logic. Other consumers (Bulk Archive, future CI integration) also need compliance checks.

### 3. Archive pre-flight compliance gate

**Decision:** `OpenSpecArchiveAction` will call `ComplianceService.checkCompliance()` before performing the archive. If the result has any ERROR-level findings, the archive is blocked and a dialog shows the findings with remediation guidance. If findings are WARNING-level only, the dialog shows them but allows the user to proceed.

**Rationale:** Currently archive succeeds even when specs are malformed. A pre-flight gate prevents non-compliant changes from reaching the archive, which is the point of no return in the OpenSpec lifecycle.

**Alternatives considered:**
- *Warn but always allow archive*: Rejected — defeats the purpose of compliance enforcement. The user already has the option to disable strict mode in settings.
- *Auto-fix compliance issues before archive*: Rejected — auto-fixing specs is dangerous. The user must review and accept changes.

### 4. Compliance observability via notification group and status chip

**Decision:** Register a new `OpenSpec.Compliance` notification group in `plugin.xml` (display type: STICKY_BALLOON). Add a compliance status indicator to `WorkflowActionPanel` — a colored chip next to the pipeline that shows "Compliant" (green), "Issues found" (yellow, with count), or "Not checked" (gray). Clicking the chip runs the compliance check and shows results in a dialog.

**Rationale:** Compliance status must be visible without the user having to run a manual action. The existing pipeline chips show artifact status but not overall compliance. A sticky balloon ensures compliance failures aren't dismissed accidentally. The status chip provides at-a-glance visibility directly in the workflow panel where the user is already working.

**Alternatives considered:**
- *Use the existing Validation notification group*: Rejected — compliance is broader than validation (includes sync readiness). Mixing them would dilute the signal.
- *Add a separate Compliance tab to the tool window*: Rejected — over-engineered for v0.3.0. A status chip + dialog is sufficient. A tab can be added later if compliance grows in complexity.

### 5. Spec-sync post-merge validation

**Decision:** After `SpecSyncService.applySync()` writes merged content to main spec files, run `BuiltInValidator.validateSpecFile()` on each affected main spec. If the merged result has validation errors, the sync reports them as warnings (the merge itself succeeds, but the user is told the merged spec needs manual correction). In strict mode, MODIFIED operations targeting nonexistent requirements produce an error and block the sync.

**Rationale:** Currently, spec-sync can produce invalid main specs — e.g., a MODIFIED operation that updates a requirement's text but removes its scenarios. Validating after merge catches this. Making strict mode fail on missing targets prevents silent no-ops when the user modifies a requirement that was renamed or removed.

**Alternatives considered:**
- *Block sync entirely if merged result has errors*: Rejected — too disruptive. The merge may be mostly correct with one minor formatting issue. Reporting as warnings lets the user fix it.
- *Validate before merge (dry-run)*: Desirable but complex — requires simulating the merge in memory and validating the projection. The current `computeSync()` already does the projection; connecting it to validation is feasible but adds complexity. Deferred to a follow-up if post-merge validation proves insufficient.

## Risks / Trade-offs

**[BREAKING: stricter validation rejects previously-passing content]** → The proposal acknowledges this. Specs generated by the plugin's own AI pipeline already comply. Manual or externally-generated specs may need updates. **Mitigation:** Clear error messages with specific guidance (e.g., "Requirement 'X' has no scenarios — add at least one `#### Scenario:` block"). Strict mode setting remains available for users who want warnings-as-errors; the new errors are the baseline.

**[ComplianceService depends on multiple services]** → It composes BuiltInValidator, VerificationService, and SpecSyncService. A bug in any of them affects compliance results. **Mitigation:** ComplianceResult is structured per-category, so a failure in one area doesn't mask results from others. Each category is independently testable.

**[Spec-sync post-merge validation may produce false positives]** → If the existing main spec already has issues, the post-merge validation will surface them even though the sync didn't cause them. **Mitigation:** The validation report distinguishes "pre-existing issues in main spec" from "issues introduced by this sync" by comparing pre-merge and post-merge validation results.

## Deferred to Backlog

1. **Strict-mode revert on post-merge validation errors.** If spec-sync produces a merged main spec that fails validation in strict mode, the current design would revert to pre-merge content. This requires saving file content before sync and restoring on failure. Deferred because it adds complexity to the sync flow and the non-strict behavior (report as warnings) is sufficient for v0.3.0.

2. **Bulk Archive compliance integration.** `BulkArchiveDialog` currently checks artifact completeness and spec conflicts but does not run the full compliance check per change. Deferred because it could make the dialog slow for many changes, and individual archive compliance gates cover the same ground.
