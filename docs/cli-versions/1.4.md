# OpenSpec CLI 1.4 — Feature Delta & Plugin Impact (vs 1.3)

> **Maintenance: Reference** — Per-CLI-version feature-delta analysis, sourced from upstream OpenSpec (Fission) documentation and cited. Reviewed when a new CLI line ships or when a plugin decision needs to re-check the 1.3→1.4 delta. **Last reviewed: 2026-07-04.** For the plugin's live support matrix see [OpenSpec Client Coverage](../openspec-support.md); for the concept model see the archived coordination-surfaces spec.

Upstream project: **OpenSpec by Fission** — npm [`@fission-ai/openspec`](https://www.npmjs.com/package/@fission-ai/openspec), repo [`github.com/Fission-AI/OpenSpec`](https://github.com/Fission-AI/OpenSpec). Every upstream claim below is cited to a source in [§6](#6-sources); claims that could not be confirmed upstream are quarantined in [§5](#5-open-questions--to-verify) and not asserted as fact.

Version line covered: `1.3.0` (2026-04-11) → `1.3.1` (2026-04-21) → **`1.4.0` (2026-06-01)** → `1.4.1` (2026-06-03). Confirmed via the npm version list and the upstream CHANGELOG headers. `1.5.0` (2026-06-28) is out of scope here except where it retires a 1.4 feature.

---

## 1. Summary

OpenSpec **1.4.0** is, in its *headline* changelog, a modest release: two new skills-only assistant integrations (Kimi CLI, Mistral Vibe), parser/validation robustness fixes (case-insensitive requirement headers, clearer validation hints), a shell-completion fix, and inclusion of the sync workflow in the default core profile. The **large** 1.4-line change is not in the headline bullets at all: the **workspace / context-store / initiative** coordination model and the **`workspace-planning`** schema shipped as a *beta* on the 1.4 line (developed in upstream changes archived 2026-05-04 → 2026-05-14, i.e. between 1.3.1 and 1.4.0, and documented under a `workspaces-beta/` docs tree). This beta is what the plugin's Coordination tab targets. It was short-lived: **1.5.0 removed it**, replacing "the workspace and initiative model" with stores. So relative to 1.3, the durable 1.4 delta is "a beta coordination layer arrived (and was gone one minor later)," layered on top of small, permanent parser/tooling improvements.

## 3. Per-feature detail

### 3.1 Kimi CLI and Mistral Vibe support (Introduced)
**Mechanics.** `openspec init` gains two additional skills-only targets: Kimi CLI (`.kimi/skills/`) and Mistral Vibe (`.vibe/skills/`), joining the existing roster of supported assistants (CHANGELOG 1.4.0, PRs #1003, #1154).
**Plugin assessment.** N-A. This plugin *is* the JetBrains-IDE integration; onboarding third-party CLI assistants is out of its scope. No UI implication.
**Consumer how-to.** Run `openspec init` in a terminal and select the assistant; unrelated to the plugin.

### 3.2 Parser / validation robustness (Modified)
**Mechanics.** Requirement headers now parse regardless of capitalization, so a `### requirement:`-style header no longer fails parsing (CHANGELOG 1.4.0, PR #1154). Separately, `openspec validate` now gives a targeted hint when a requirement carries `SHALL`/`MUST` only in its header, telling the author to move the keyword onto the requirement body line rather than emitting the generic error (CHANGELOG 1.4.0, PR #1154).
**Plugin assessment.** Supported (mirrored built-in via the `validator-parity-with-cli-1-4` change). The plugin has its own built-in delta-spec inspection/validation (it does not shell out to `openspec validate` for inline checks), so these two CLI-side improvements were mirrored explicitly: requirement headers are recognized case-insensitively through one shared pattern used by the validator, both inspections, tree parsing, and spec sync (sync writes stay canonical `### Requirement:`), and a keyword-only-in-header requirement gets the CLI's targeted "move the keyword onto the requirement body line" diagnostic with an inspection quick-fix.
**Consumer how-to.** The CLI and the plugin now agree on header casing and on the keyword-placement hint; if they ever disagree again, trust the CLI and file it as a plugin parity gap.

### 3.3 Zsh completions on oh-my-zsh (Modified, bugfix)
**Mechanics.** Fixes shell-completion installation under oh-my-zsh's `compinit` (CHANGELOG 1.4.0, PR #1154). Terminal-only.
**Plugin assessment.** N-A. No plugin surface.

### 3.4 Sync workflow in the default core profile (Introduced/Modified)
**Mechanics.** New installs now generate the `/opsx:sync` skills and commands by default, i.e. the delta→main spec sync workflow is part of the core profile rather than opt-in (CHANGELOG 1.4.0, PR #1030).
**Plugin assessment.** Yes. The plugin already ships a **built-in** delta→main spec sync (plus an AI skill) that works regardless of CLI version, so it is at or ahead of this default. No new work required; the change simply aligns fresh CLI installs with what the plugin already assumes is available.
**Consumer how-to.** Use the tool-window sync action (or the `openspec-sync-specs` AI skill). The CLI change means new projects also get `/opsx:sync` for terminal use.

### 3.5 Coordination model beta: workspace / context-store / initiative + `workspace-planning` schema (Introduced, beta)
**Mechanics (cited).** On the 1.4 line OpenSpec gained a cross-repo/cross-area planning layer:
- **`workspace`** — set up a "planning home" and link existing repos/folders so OpenSpec can see them without first creating a change. Confirmed subcommands from the upstream archived proposal: `workspace setup` (incl. `--no-interactive --name … --link …`), `workspace list`/`ls`, `workspace link`, `workspace relink`, `workspace doctor` (upstream archive `2026-05-06-workspace-create-and-register-repos/proposal.md`).
- **`context-store`** and **`initiative`** — a shared coordination store holding initiatives with a lifecycle status. Confirmed as concepts/commands by the upstream repo's own `openspec/initiatives/context-store-and-initiatives/` initiative (with work-items "add-context-store-foundation", "ship-initiative-mvp") and by the v1.5.0 changelog explicitly "replacing the workspace and initiative model" (CHANGELOG 1.5.0). The exact `context-store`/`initiative` subcommand surface is not fully quoted in a single upstream doc I could reach — see §5.
- **`workspace-planning` schema** — a package-provided built-in schema. The upstream schema-resolution spec states `openspec schemas` output "SHALL include `workspace-planning`" as package-provided, resolved with project→user→package precedence, defaulting to `workspace-planning` inside a workspace-planning home and overridable via `--schema` (upstream archive `2026-05-14-workspace-change-planning/specs/schema-resolution/spec.md`).

**Timing / why it's not in the headline changelog.** These features were developed in upstream changes archived 2026-05-04 through 2026-05-14 — after 1.3.1 (2026-04-21) and before 1.4.0 (2026-06-01) — and were documented under a `workspaces-beta/` docs tree, i.e. shipped as a **beta**. That is why the 1.4.0 headline changelog bullets do not announce them, even though the 1.4.0 *patch* note ("preserve workspace planning detection …", PR #1111) and the 1.4.1 note (PR #1165, `.openspec-workspace/view.yaml`) both operate on this machinery — proving it was live on the 1.4 line.

**Lifespan.** Short. **1.5.0 removed** this model in favor of stores/worksets ("Stores … replacing the workspace and initiative model", CHANGELOG 1.5.0). So the whole coordination layer exists only in the `[1.4.0, 1.5.0)` window.

**Plugin assessment.** Yes (Partial), and this is where the 1.4 delta actually lands for the plugin. The plugin's **Coordination tab** targets exactly this: it resolves the three collections via `workspace list --json`, `context-store list --json`, `initiative list --json` (plus `doctor`) **only when the detected CLI is in `[1.4.0, 1.5.0)`**, with a built-in fallback that reads the global data dir directly, and Full-tier write actions (New Initiative, Set Up Context Store, Set Up Workspace) that are **self-retiring** on a 1.5 upgrade. The plugin's own spec (`coordination-surfaces`) already encodes this window gate and the on-disk fallback, and the `CliVersion.inRange` helper implements the `[1.4.0, 1.5.0)` check.
**Consumer how-to.** With a 1.4.x CLI installed, the Coordination tab lists workspaces (with resolution health), context stores (id/root + doctor health), and initiatives (id/title/status badge); Full tier enables the three write actions and initiative-artifact navigation. Below 1.4 or on 1.5+, the tab degrades to read-only Awareness (or Hidden). For anything the tab doesn't surface, use the CLI directly in the `[1.4.0, 1.5.0)` window.

## 4. Decisions influenced

1. **Do not invest new UI in the 1.4 coordination beta beyond what already exists.** It is a beta that upstream removed one minor release later (1.5.0). The plugin's existing window-gated, self-retiring Coordination tab is the right amount of investment; broadening per-mode authoring UX for `workspace-planning` specifically is low-value since the schema is gone in 1.5. Effort on coordination should target the **stores/workset** (1.5) model, not deepen 1.4.
2. **Mirror the two parser/validation improvements into the built-in validator (§3.2).** Case-insensitive requirement headers and the SHALL/MUST-in-header hint are permanent, cheap, and prevent the plugin's inspections from disagreeing with a spec the CLI accepts. These are the only *durable* 1.4 changes with a real plugin surface.
3. **No action needed for Kimi/Mistral/zsh/sync-default.** Three are out of scope (other assistants, shell completion) and the sync default is already met by the plugin's built-in sync.
4. **Keep the 1.4 line's floor and window semantics as-is.** The evidence confirms 1.4 is the correct floor for coordination reads/writes and `[1.4.0, 1.5.0)` is the correct window; no re-gating warranted.

## 5. Open questions / to verify

- **`set` command as a "1.4 addition."** The plugin's `openspec-support.md` version-support note lists "`workspace` / `context-store` / `initiative` / `set` commands" as 1.4 additions. I found **no upstream confirmation of an `openspec set` command** in the 1.4 changelog or the reachable docs. Treat `set` as **unverified**; confirm empirically with `openspec --help` on a 1.4.x CLI before relying on it.
- **Exact `context-store` / `initiative` subcommand surface.** Confirmed as *concepts/commands* on the 1.4 line (upstream initiative directory + v1.5.0 replacement note), and the plugin invokes `context-store list --json` / `context-store doctor` / `initiative list --json` / `initiative create`. I could not pull a single upstream doc that enumerates every `context-store`/`initiative` subcommand verbatim (the `workspaces-beta/` docs tree that would have is gone from `main`, replaced by `stores-beta/`). The plugin's calls should be re-confirmed against a real 1.4.x CLI, or against the archived `workspaces-beta/` docs at a 1.4.x git tag.
- **Whether `context-store`/`initiative` shipped exactly in 1.4.0 vs a later 1.4.x point.** The archived development window (May 4–14) and the 1.4.0/1.4.1 patch notes place the model on the 1.4 line, but the precise point release that first exposed each command is not pinned by the headline changelog (beta feature). Not load-bearing for the plugin (it gates on the whole `[1.4.0, 1.5.0)` window), but flagged for honesty.

**Discrepancy found — plugin belief vs upstream (RENAMED delta type).** `openspec-support.md` states `RENAMED` "tracks the 1.4 client addition." This is **incorrect**: upstream introduced all four delta types — `## ADDED / MODIFIED / REMOVED / RENAMED Requirements` — together in **1.0.0** (CHANGELOG 1.0.0, "From Text Merging to Semantic Spec Syncing"). `RENAMED` is not a 1.4 addition; it predates the plugin's 1.3.0 floor entirely. The plugin implements RENAMED built-in and works at the floor, so behavior is unaffected — but the doc's "1.4 client addition" attribution should be corrected.

**Reconciliation of plugin belief vs upstream (coordination model).** The plugin's belief that "`workspace-planning` schema and the `workspace`/`context-store`/`initiative` commands are 1.4 additions" is **substantially correct** — with the nuance that they landed as an *undocumented-in-headline beta*, not as advertised 1.4.0 features. The plugin's empirical "compare CLI 1.3.1 ↔ 1.4.0 command surfaces" method is exactly why it caught them despite the silent changelog; that method is sound and should be kept.
