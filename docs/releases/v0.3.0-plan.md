### 6. Forgejo runner verifyPlugin fix (release blocker)

**Status:** documented in `project_forgejo_runner_verify_cancellation` memory; root cause unknown.

**The problem:** `verifyPlugin` job cancels mid-execution on the Forgejo runner. Observed twice during v0.2.10 prep. GitHub Actions runs the same code without issue (verify takes ~9–14m). Likely causes (in order):
1. Runner job-duration limit lower than verify needs (15+ min).
2. Memory pressure / OOM during multi-IDE Plugin Verifier load (~10 GB+ platform downloads).
3. Runner-side concurrency rules cancelling jobs on edge cases.

**Possible fixes:**
- (a) Raise the runner's job-duration limit (host config on `geek`).
- (b) Increase runner host memory.
- (c) Split `verifyPlugin` to run fewer IDE versions per Forgejo run.

**Path to release-ready:** investigate runner logs on `geek:222`; pick the smallest viable fix; ship. If unresolvable in time, document it formally: "Forgejo CI is non-canonical for verify; GitHub CI is the source of truth for Marketplace publish."

**Refs:** Forgejo #212, Plane OS-223 (high priority). If the fix needs runner-host changes, file an inter-agent issue against `johnb/homelab-admin` via `mcp__homelab__forgejo_open_issue` per [[reference_feedback_to_homelab]].

### 7. CLI floor bump to 1.3.0 (precondition for #8)

**Status:** filed 2026-06-11. Not started.

**Why:** designing the plugin around 4-year-old CLI baselines is engineering tax for a near-zero user population. Drop pre-1.3 and the V1_0/V1_1 baselines become deletable, the validator simplifies, and the BC matrix for OS-218 collapses to a manageable 3-column shape.

**Ships:**
- Delete `VersionSupport.V1_0` and `V1_1` enum entries; `V1_2` becomes the only baseline.
- Simplify `BuiltInValidator` paths that branch on V1_0 / V1_1.
- Update `SchemaService.isSchemaSupported` floor from 1.2.0 to 1.3.0.
- Plugin startup notification when CLI < 1.3.0 detected, pointing to upgrade command.
- README + Marketplace listing: update minimum CLI to 1.3.0.
- CHANGELOG entry under v0.3.0 marked **BREAKING**: "Minimum supported OpenSpec CLI is now 1.3.0."

**Path to release-ready:** one OpenSpec change, ~one session of work. Low risk — pre-1.3 users continue to function on the plugin's built-in no-CLI fallback paths.

**Refs:** Forgejo #214, Plane OS-224.

### 8. IDE surfaces for OpenSpec 1.4.x feature areas

**Status:** revised scope 2026-06-11 — was OUT, now IN with a more achievable framing.

**The reframe:** research showed that workspace / context-store / initiative state lives entirely in user-global XDG paths under `~/.local/share/openspec/`. **The repo on disk contains nothing 1.4.x-specific.** The plugin can read this state directly via snakeyaml without invoking the CLI. This unlocks a 3-state UX model (Hidden / Awareness / Full) rather than the binary on/off pattern in `SchemaService.isSchemaSupported`. Captured as durable insight in [[project_xdg_state_directly_readable]].

**Ships:**
- `WorkspaceMembershipService` + parallel for context-stores / initiatives — pure filesystem readers, no CLI invocation.
- Read-only **Awareness** UI for any CLI version: status-bar widget "Linked to workspace `foo`"; tool-window section listing context-stores / initiatives.
- **Full** action surfaces gated on CLI ≥ 1.4.0: workspace setup / link, context-store register, initiative create.
- Hidden state for everything when no relevant disk state exists.

**Out of scope (this item):** `openspec init --profile` flag wiring — not a 1.4.x feature (existed since 1.2). Worth its own small ticket later or rolling into profile-ui-cli-alignment.

**Path to release-ready:** biggest single item in v0.3.0. Estimated 2–3 sessions of work split across discovery, awareness, and full-action layers. Depends on **#7 (CLI floor bump)** landing first to simplify the BC matrix.

**Refs:** Forgejo #207, Plane OS-218. Body now contains the full 3-state matrix and detection pattern.

**Risk:** highest of the IN items. The most likely candidate to slip to v0.3.1 if v0.3.0 needs to ship sooner. If slipping, the awareness layer alone (read-only YAML scan + status-bar widget) is a defensible smaller scope.

### 9. Schema architecture refactor (low priority)

**Status:** filed 2026-06-11. Not started.

**Why:** same architectural lesson as profile-ui-cli-alignment, applied to schemas. `VersionSupport.getValidSchemas()` hardcodes the schema set; users who run `openspec schema fork spec-driven my-team-flow` get spurious `change-schema-incompatible` warnings from the plugin even though the CLI accepts the schema. Upstream's Zod allows any non-empty string; the plugin is being more restrictive than the CLI.

**Ships:**
- `VersionSupport.getValidSchemas()` reads `openspec schemas --json` at runtime (cached per session).
- Settings → Default schema combo populates from `SchemaService.listSchemas()` instead of any hardcoded list.
- No-CLI fallback: accept the two known built-in schemas (`spec-driven`, `workspace-planning`); validator becomes permissive (warn, not error) on unrecognized schemas.

**Path to release-ready:** one OpenSpec change, ~one session. **Lowest priority of the IN items.** First candidate to slip to v0.3.1 if scope pressure mounts.

**Refs:** Forgejo #215, Plane OS-225.
