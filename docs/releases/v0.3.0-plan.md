### 4. ScaffoldingService bug fix

**Status:** filed (Forgejo #208 / Plane OS-219). Pre-existing latent bug, surfaced by `workspace-planning` becoming user-selectable.

**The bug:** `ScaffoldingService.initBuiltIn` at line 140 hardcodes `TemplateProvider.configYamlTemplate("spec-driven", …)` regardless of `OpenSpecSettings.getDefaultSchema()`. Users who set a non-`spec-driven` default schema see `spec-driven` written anyway when the built-in init path runs.

**Fix sketch (from the issue):** thread `OpenSpecSettings.getInstance(project).getDefaultSchema()` through, with `"spec-driven"` fallback when empty; mirror the existing `getEffectiveVersion()` pattern at line 154. Add a `ScaffoldingContractTest` covering it.

**Estimated:** small (one-session change).

### 5. CI caching quick wins

**Status:** scoped in `project_ci_caching_optimization` memory; not started.

**Ships:**
- Add `concurrency:` blocks to `.github/workflows/build.yml`, `.github/workflows/release.yml`, `.forgejo/workflows/build.yaml`. Pattern: `concurrency: { group: ${{ github.ref }}, cancel-in-progress: true }`. Prevents the "old + new CI run in parallel" waste observed during v0.2.10's retrigger commit.
- Bump deprecated action versions in the Forgejo workflow: `actions/cache@v3 → v4`, `actions/upload-artifact@v3 → v4`. (GitHub workflow already on v4.)
- Enable Gradle build cache via `gradle.properties`: `org.gradle.caching=true`.

**Estimated:** ~15-minute change, zero risk. Could be a single small OpenSpec change.

### 6. Forgejo runner verifyPlugin fix (release blocker)

**Status:** documented in `project_forgejo_runner_verify_cancellation` memory; root cause unknown.

**The problem:** `verifyPlugin` job cancels mid-execution on the Forgejo runner. Observed twice during v0.2.10 prep. GitHub Actions runs the same code without issue (verify takes ~9–14m). Likely causes (in order):
1. Runner job-duration limit lower than verify needs (15+ min).
2. Memory pressure / OOM during multi-IDE Plugin Verifier load (~10 GB+ platform downloads).
3. Runner-side concurrency rules cancelling jobs on edge cases.

**Possible fixes:**
- (a) Raise the runner's job-duration limit (host config on `geek`).
- (b) Increase runner host memory.
- (c) Split `verifyPlugin` to run fewer IDE versions per Forgejo run.

**Path to release-ready:** investigate runner logs on `geek:222`; pick the smallest viable fix; ship. If unresolvable in time, document it formally: "Forgejo CI is non-canonical for verify; GitHub CI is the source of truth for Marketplace publish."

**Refs:** open a Forgejo issue against `johnb/homelab-admin` if the fix needs runner-host changes (use `mcp__homelab__forgejo_open_issue` per `reference_feedback_to_homelab`).
